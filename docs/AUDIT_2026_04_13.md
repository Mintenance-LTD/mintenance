### Top 10 "must fix before any production deploy"

1. **[P0 DB]** 8 tables have RLS enabled but zero policies (ab_arms, ab_calibration_data,
   cdn_upload_queue, learning_model_versions, ml_metrics, performance_metrics, production_metrics,
   rate_limits) — §6.2
2. **[P0 DB]** `announcements.admin_manage_announcements` policy uses
   `USING (true) WITH CHECK (true)` — unrestricted ALL operations — §6.3
3. **[P0 DB]** `spatial_ref_sys` has RLS disabled (PostGIS system table exposed via PostgREST) —
   §6.3
4. **[P0 Lifecycle]** Homeowner approval is auto-triggered after 7 days without explicit gate on
   `completion_confirmed_by_homeowner` — spec/code mismatch, funds can release without user action —
   §7.10
5. **[P0 Shared]** `packages/ai-core` uncommitted test uses wrong `limits` field names vs
   `CostLimit` type — blocks CI — §5.3
6. **[P0 Web FE]** Bulk delete on `JobsBulkActionsSection.tsx:100` bypasses CSRF — §1.3
7. **[P0 Web FE]** `TryMintAIClient.tsx:94` sends unvalidated base64 images (no type/size cap) —
   §1.3
8. **[P0 Supply-chain]** 10 files still using direct `createClient` from `@supabase/supabase-js` —
   RLS bypass / service-key risk — §2.2
9. **[P1 Lifecycle]** Escrow release uses `job.contractor_id` at release time, not `escrow.payee_id`
   at creation — if contractor reassigned mid-job, funds go to wrong account — §7.10
10. **[P1 Web BE]** `PATCH`/`DELETE` on `/api/jobs/[id]/route.ts:16-22` are NOT wrapped with
    `withApiHandler` — bypass rate limit and structured error middleware — §2.4

### 7.10 Phase 10: Payment release — **5 critical findings**

**LFC-P0-1** — **Homeowner approval is auto-triggered, not gated.**
[`apps/web/lib/services/agents/escrow/evaluate.ts:109-126`](apps/web/lib/services/agents/escrow/evaluate.ts#L109-L126)

```ts
if (!escrow.homeowner_approval) {
  const autoApprovalEligible =
    await HomeownerApprovalService.checkAutoApprovalEligibility(escrowId);
  if (!autoApprovalEligible) return null;
  await HomeownerApprovalService.processAutoApproval(escrowId); // auto-approves
}
```

After 7 days, `homeowner_approval` is flipped to true **without the homeowner ever acting**. This
contradicts the CLAUDE.md spec which says "payment release requires homeowner approval". The 7-day
rule was intended as a safety net, but the code treats it as an automatic substitute. **There is no
explicit gate on `completion_confirmed_by_homeowner`.**

**Fix**: add explicit check:
`completion_confirmed_by_homeowner === true OR (job.completed_at + 7d < now() AND photo_verification_passed AND no active dispute)`.
Log all auto-approvals distinctly from manual approvals.

**LFC-P1-1** — **Escrow released to wrong contractor after reassignment.**
[`apps/web/lib/services/escrow/EscrowAutoReleaseService.ts:190`](apps/web/lib/services/escrow/EscrowAutoReleaseService.ts#L190)

```ts
const contractorStripeAccountId = contractorStripeMap.get(job.contractor_id);
```

Uses the **current** `job.contractor_id`, not the `escrow.payee_id` recorded at escrow creation. If
a job is reassigned mid-flight, the new contractor receives funds bid by the original.

**Fix**: lock `escrow.payee_id` at creation and assert match at release:
`assert(job.contractor_id === escrow.payee_id) || abort`.

**LFC-P1-2** — **Spec vs code mismatch on manual approval requirement.** The
`completion_confirmed_by_homeowner` column exists and is set in `/api/jobs/[id]/confirm-completion`,
but auto-release never reads it. Either the column is load-bearing (and auto-release should check
it) or it's advisory (and the spec should be rewritten). Currently it's ambiguous.

**LFC-P1-3** — **Bid acceptance race condition UX.** DB unique constraint correctly serializes, but
concurrent callers see 23505. Add optimistic check before UPDATE to return cleaner 409.

**LFC-P1-4** — **Asymmetric contract signing notification.**
[`/api/contracts/[id]/accept/route.ts:130-150`](apps/web/app/api/contracts/[id]/accept/route.ts#L130-L150)
notifies homeowner when contractor signs but not the other direction (unless both signed).
